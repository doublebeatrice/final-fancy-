# Codex Handoff - 2026-04-23

这份文档给下一任 Codex 直接接手当前广告自动化项目用。先读本文件，再读 `README.md`、`AGENT.md`、`task.md` 和 `docs/Q2_AD_OPS_PLAYBOOK.md`。

## 当前总边界

当前架构已经定为：

- Codex 是唯一 AI 决策入口。
- 插件面板不内置 AI provider，不在面板里做策略判断。
- 面板只负责抓数、导出结构化数据、可视化、调用真实接口、展示执行结果。
- Codex 在程序外读取快照、理解库存/广告/历史/节气/Q2/KPI，然后生成 action schema。
- 执行层只负责 schema 校验、接口执行、回查、便签、summary。
- 不允许执行层偷偷保留第二套策略函数来冒充 AI 决策。

## 运行前必须确认

每次全量测试或执行前，需要打开调试浏览器，并由用户手动确认两个后台已经登录：

- `https://adv.yswg.com.cn/`
- `https://sellerinventory.yswg.com.cn/`

如果没有用户确认登录状态，快照可能只抓到库存、抓不到广告数据。

常用入口：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\execute\open_debug_browser.ps1
node scripts\execute\export_snapshot.js data\snapshots\latest_snapshot.json
```

执行前先 dry-run：

```powershell
$env:DRY_RUN='1'
```

```powershell
Remove-Item Env:\DRY_RUN -ErrorAction SilentlyContinue

```

## 当前策略口径

用户最新要求是利润导向、Q2导向、应调尽调、应建尽建：

- 不要只看当前 ACOS。
- 流量期、库存可承接、价格有优势、历史广告有基础时，可以更激进。
- 低销量或无销量不等于产品不行，可能只是广告覆盖不足。
- 覆盖率也要纳入考核。
- Q2重点：listing、广告策略/预算、库存管理。
- 节气产品、即将滞销、展示点击不足、有库存的 SKU，要优先考虑加覆盖、开历史广告、涨竞价、建 SP 广告。
- SB 暂时以调整为主，SB 新建仍未放开，因为真实创建接口没验证。
- 新建 SP 广告现在无需审核，但要基于库存、利润、Q2、节气、低覆盖、滞销风险等证据。

后续追加的业务口径：

- 用户明确要求“说人话”，不要用抽象术语解释策略。
- 不要只按 ACOS 调整，`同比下滑` 必须纳入 AI 决策。
- 但同比不是死规则，不能写成固定 if/else；Codex 需要自己从实时盘面找规律。
- 对于高销量或高承接 SKU，允许更激进。
- 对于高货值、大件、滞销费高、5 月是最后主卖窗的 SKU，即使同比下滑，也可能要激进推而不是收缩。
- 当前用户明确点名以下 SKU 需要按“5 月窗口去库存”口径激进推：
  - `DN3482`
  - `DN3049`
  - `DN2685`
  - `DN2684`
  - `DN2683`
  - `DN2437`
  - `DN1656`
  - `DN2108`
  - `DN1655`

本轮对同比问题的处理：

- `extension/panel.js` 已把库存同比字段继续透传到 product card：
  - `yoySales`
  - `yoySalesPct`
  - `yoyUnitsPct`
  - `yoyRank`
- `src/ai_decision.js` 已将这些字段带入 AI context。
- `scripts/generators/generate_profit_adjust_schema.js` 已加入同比下滑恢复逻辑，避免只按 ACOS 机械收缩。
- 但用户最新要求是不把“同比下滑”写成固定规则，而是让 Codex 结合库存压力、窗口期、利润、覆盖、历史广告基础动态决策。

便签口径需要注意：

- 不要写成单纯“ACOS 高/低所以调价”。
- 流量期应写明：流量期、库存承接、价格优势、历史广告基础、当前覆盖不足、目标是提升展示/点击/转化。
