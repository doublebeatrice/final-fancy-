项目已经接近“工程闭环可执行”，但还没有达到“自然语言一句执行今日全量广告运营”的无人值守阶段；当前更准确的状态是 外部 AI 决策 + 本地确定性执行引擎。

1. 顶部总览

项目定位

  • 不是内嵌 AI 面板，而是广告运营执行底座。
  • 面板负责采集、展示、桥接；Codex 负责理解业务、输出 action schema。
  • 执行层强调确定性：只执行 schema，不在执行器里偷偷做策略。

是否接近「一句执行」

部分接近

  • 接近的是：底层执行链路已经成形。
  • 不接近的是：自然语言→全量策略→自动落地 仍需外部 Codex 人工触发与审核边界。
  • 阻塞点主要不是“不会调 API”，而是“口径统一、动作范围、回查可靠性、人工前置条件”。
判断：项目已经不是 demo，也不是单点脚本；它已经是一个可跑的广告运营执行平台雏形，但离“今天一句话让 AI 做完全部广告运营”还差最后一层调度自治与风险收口。

3. 数据源地图

adv 系统接口

主数据源
  • `/keyword/findAllNew`：SP keyword / SP auto / SP manual / SB keyword / SB target。
  • `/product/adSkuSummary`：SKU 30 天广告汇总与 delta。
  • `/product/chart`：SKU 展示 / 点击时间趋势；已纳入节气品、季节品、老品补量判断依据。

  • `/advTarget/findManualProductTarget`、`/customerSearch/targetFindAll`：SP ad group 内部详情。
  • `/advProduct/all`、`/campaignSb/findAllNew`：SP product ad / SB campaign 管理视角。

sellerinventory 系统接口

第二主链路
  • `/pm/formal/list`：inventory 总盘。
  • `/pm/formal/update`：便签写入。
  • `/pm/sale/getBySeller`：个人销售数据。
  • 节气接口：活跃季节 / 标签上下文。

本地配置 / 缓存 / 导出

支撑层
  • `data/snapshots/*.json`：快照、回查、summary、coverage。

  • `data/product_profiles.json`：商品画像缓存。

新增规则：`/product/chart` 现在是正式判断依据之一。对节气品、季节品、老品补量场景，如果当前处于热卖窗口且库存可承接，同时 chart 显示展示 / 点击绝对值在下滑，则系统判断应优先考虑“补流量 / 加投”，而不是仅按历史 `降投` 备注继续压量。

6. 执行日志、回查、冷却机制、失败结构化返回

执行日志

  • `data/snapshots/auto_run_YYYY-MM-DD.log`
  • `execution_verify_*.json`：事件级验证
  • `execution_summary_*.json`：SKU 级汇总
  • `execution_coverage_*.json`：覆盖面状态

冷却与回查

  • `data/adjustment_history.json` 记录成功历史。
  • 7 天窗口阻止重复触碰或系统近期已调对象。
  • 先增量刷新，失败后全量重抓，避免 stale cache 假成功。

结构化失败

  • create / toggle 都有 structured result。
  • 执行后区分 `manual_review / blocked / not_landed / failed`。
  • 便签写入失败按 SKU 返回，不会吞错误。

7. 当前最大风险

风险 等级 现状 为什么重要
接口成功但未落地 高 历史上真实出现过；当前已通过 verify 将其显式归类为 `not_landed`。 这是广告自动化里最危险的“假成功”；虽然现在已不再静默，但仍是首要可信度风险。
token / csrf / 登录态失效 高 整个系统强依赖 debug Chrome、登录态、面板桥。 一旦浏览器未登录或 token 失效，抓数和写入都可能退化成 HTML 页面或空数据。
数据口径不一致 高 inventory、adSkuSummary、adProductData、panel 聚合、同日 fallback 口径并存。 会直接影响“该不该调”“调大还是调小”的判断可信度。
fallback 假成功 中高 现已明显改善，但创建广告仍可能落在 `created_pending_visibility`。 如果可见性慢或刷新范围不对，会误把“延迟落地”与“没落地”混在一起。
模块命名空间串用 中高 SB 与 SP 端点、placement 字段、candidate 与 executable entity 曾出现串路由问题。 这是最典型的“代码看着通了，业务实际改错对象”的工程型风险。
inventory aid 缺失 中 便签写入仍可能因 inventory 记录缺失而失败。 不影响广告改价本身，但会破坏运营闭环留痕和追责链。

10. 下一步路线图

优先级 事项 业务价值 风险 验证方式
P1 把“今日全量运营”收束为单入口命令 从多脚本拼装变成标准操作，降低认知成本。 容易把前置条件藏起来。 要求一条命令串起：快照→schema 校验→dry-run→execute→report，并对未登录直接报错。
P1 补齐可执行实体矩阵 把“开关广告 / 七天未调 / create”从半完成变成稳定面。 错误路由会造成误改。 对 `campaign / adGroup / productAd / sbCampaign / candidate` 建一个显式覆盖表并做小样本真机回归。
P1 建立统一落地验证协议 继续压缩“接口成功但未落地”的灰区。 不同 endpoint 的回查口径不同。 每种 action 定义 `expected_state/value` 与 `verify source`，禁止无回查成功入 history。
P2 把 budget / placement 变成正式运营模块 开始从“调 bid”升级到“调 campaign 结构变量”。 预算和 placement 对流量影响更大。 先做受控名单、固定 measurement window、归因复盘。
P2 收敛数据口径与质量标记 避免 AI 在混合口径上做出错误策略。 需要改 snapshot 元信息结构。 每个字段标注来源接口、时间窗、是否 fallback、是否 same-day borrow。

不要碰哪些地方

  • 不要把 AI provider 再塞回 panel。
  • 不要让执行器自己做策略 fallback。
  • 不要硬编码 JWT / XSRF / Inventory-Token。
  • 不要把 review-only 动作偷偷放进 auto-execute。

修改后必须回归

  • snapshot 导出是否完整。
  • SP / SB bid 执行与 verify 是否一致。
  • note 写入是否只重试失败项。
  • `not_landed / conflict / failed` 是否仍然分开。
  • named SKU 最小接口路径是否仍可用。
交接判断：这个仓库最有价值的不是某个规则函数，而是它已经把“广告运营动作”拆成了可验证的工程链。接手时应优先维护链路可信度，而不是先继续堆策略。
